Data Logging Best Practices

How to design reliable hive data logging—what to measure, how to sample, calibrate, store, secure, and use your sensor data without drowning in it.

← Back to Data Sensing

Why log hive data?

Good logs turn scattered sensor readings into decisions you can trust. Temperature and humidity show brood conditions and ventilation; weight traces nectar flows and winter consumption; acoustics reveal stress and swarm preparation. Logging adds three things your memory cannot: consistent timing, auditability, and the ability to compare across colonies and seasons.

This guide focuses on robust practice, not fancy hardware. Start simple, automate the boring parts, and make sure every number has a known origin, unit, and clock.

Plan first: questions over gadgets

  • Decide what you need to know: “Are my colonies light and at risk?” “Did a treatment reduce mite stress?” Your questions dictate sensors and sampling rate.
  • Choose a minimal set: For most yards, start with internal temperature & humidity in the brood nest and a hive scale under at least one representative hive.
  • Budget for power and enclosure: A great sensor in a leaky box with dying batteries is worse than none. Weatherproof first.
  • Make the data useful: Decide where the data will live (data management) and how you’ll view it (dashboard, spreadsheet, app).

Sampling strategy and rates

Sampling more often isn’t always better; it costs power and storage and can amplify noise. Use rates that capture the biology of interest:

  • Internal temperature: every 5–10 minutes shows brood stability and anomalies without excessive drain.
  • Internal humidity: every 5–10 minutes paired with temperature; trends matter more than single points.
  • Hive weight: every 15 minutes to 1 hour balances resolution of forager returns with battery life.
  • Weather station (yard-level): every 10–15 minutes for rainfall, ambient temperature, wind.
  • Acoustics: schedule short 10–30 second recordings every hour or on a triggered event to save space.

Apply duty cycling (sensor sleeps between reads) and edge aggregation (average, min, max over the interval) to reduce raw volume while preserving information.

Sensor placement basics

  • Brood sensor: Mount just above the brood nest centerline, not touching comb or bees directly. Avoid contact with wet cappings.
  • Humidity: Shield the probe from drips and propolis; ensure airflow. Do not place against the cover where condensation collects.
  • Scale: Level the platform, avoid point loads, and keep feet off soft soil. Note any rock shims in your metadata.
  • Weather: Place rain and temperature sensors in open air, shielded from sun with a radiation shield, away from building heat.

Map each sensor’s physical position in your records. Unknown placement is a common reason data cannot be interpreted later.

Calibration and quality checks

Uncalibrated sensors lie politely. Bake in simple checks:

  • Temperature: Compare probes in a stirred ice bath (should read ~0 °C) and at room temperature side-by-side. Record offsets.
  • Humidity: Use known-salt packs (e.g., 75% RH) to verify readings annually.
  • Weight: Verify with known loads (e.g., two bricks = 5 kg). Log the test date and observed offset.
  • Clock: Ensure all devices sync time (see “Timekeeping”). Misaligned clocks make cross-hive plots misleading.

Add reasonableness filters to your logger: discard impossible spikes (e.g., ±10 kg in one minute) unless also flagged as “lifted hive” via a tilt sensor or lid-open switch.

Timekeeping and identifiers

  • UTC timestamps: Store in UTC (2025-10-25T14:03:00Z). Convert to local time only in the dashboard.
  • Stable IDs: Each colony gets a permanent colony_id; each physical box or location can have its own hive_id; each device has a device_id. Keep a lookup table.
  • Event markers: Log discrete events (feed, move, add super, treatment) with timestamp and note. They give context to data swings.

A simple, durable data schema

Use a tidy, append-only layout so any tool can read it later. One row per measurement:

timestamp_utc, apiary_id, hive_id, colony_id, device_id, sensor, value, unit, qc_flag
2025-10-25T14:03:00Z, Y01, H07, C07, D-TH-02, temp_internal, 35.2, C, ok
2025-10-25T14:03:00Z, Y01, H07, C07, D-TH-02, rh_internal,   56.0, %, ok
2025-10-25T14:03:00Z, Y01, H07, C07, D-SCALE-01, weight,     42.8, kg, ok

Keep units explicit (C, %, kg) and use qc_flag for notes like lid_open, lifted, sensor_fault.

Storage, backup, and retention

  • Local first: Log to a local file or database on the gateway (e.g., a field computer) so you don’t lose data if the network drops.
  • Daily sync: Mirror to a second location (NAS or cloud bucket). Automate with a nightly job and verify completion.
  • Retention tiers: Keep full-resolution data for 12 months, then archive downsampled summaries (5-min → hourly) to save space.
  • Versioned folders: Use dated folders or table partitions (year=2025/month=10) for fast queries and clean housekeeping.

Connectivity in real yards

Design for outages. Gateways should queue data for days and retry uploads. If using cellular, log signal strength and data usage; if using long-range radios, record packet loss rate for diagnostics.

Mark gaps explicitly (no silent flat lines). Your dashboard should draw a gap when no samples exist, not interpolate.

Power: keep it alive

  • Duty cycle: Sleep between reads and radio transmissions. Batch sends to once per 5–15 minutes.
  • Battery choice: Use cells rated for cold; mount packs in insulated boxes but allow ventilation to avoid moisture damage.
  • Solar: Right-size the panel and include a charge controller; log battery voltage to catch failures early.

Security and integrity

  • Signed firmware and updates: Prevent rogue code from hijacking devices.
  • Unique credentials per device: Avoid shared passwords. Rotate keys if a device is lost.
  • Immutable logs: Append-only storage or checksums make tampering evident; keep raw files read-only after upload.
  • See Internet Security for Smart Hives for a deeper checklist.

QA/QC: trust but verify

  • Range checks: Reject physiologically impossible values (e.g., 80 °C internal).
  • Spike detection: Flag sudden steps that coincide with no event markers.
  • Drift checks: Compare sensors across hives at night; large persistent offsets suggest calibration drift.
  • Cross-validation: Internal temp should be near brood target (~34–36 °C). If weight drops while foraging should rise, check scale feet and weather (rainwater).

Dashboards and alerts that help, not harass

  • Few, clear views: A yard overview (latest weight, temp, alerts) and a deep-dive per hive.
  • Context layers: Overlay weather, super additions, moves, and treatment events on charts.
  • Sane alerts: Alert on duration (e.g., internal temp < 30 °C for > 60 minutes) to avoid false positives.

For building, see Remote Hive Monitoring Systems and Smartphone Apps.

Metadata and documentation

Data without context is noise. Maintain a simple catalog:

  • Devices: model, firmware, calibration offsets, install date, location.
  • Hives: colony ID, queen age/line, configuration (boxes, frames), treatments.
  • Sensors: serial, placement diagram/photo, units, sampling rate.

Store this with your data so exports remain meaningful years later.

Ethics, privacy, and sharing

Yard locations, neighbor behavior, and pesticide incidents can be sensitive. If you share to community platforms (citizen science, open data):

  • Obscure exact coordinates if needed; share at the landscape scale.
  • Remove personal identifiers from notes before publishing.
  • Include a data license so others know how they can use your contribution.

Start small, iterate, and compare

Instrument one or two representative colonies first. Run for two weeks, review plots, and prune what isn’t useful. Add complexity only when you’ve proven that new data changes decisions.
